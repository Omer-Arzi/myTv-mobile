# About this app

MyTV is a personal TV-series tracking app (like TV Time). The repo root (`my-tv/`, not itself a git repo) has three subdirectories, each its own separate git repo — `cd` into the right one before working, since history/remote/`.claude/settings` are all per-subdir:

- **`server/`** — NestJS + TypeScript + PostgreSQL/Prisma backend, exposes `server/API_CONTRACT.md` as the contract the mobile app consumes. No real auth yet (hardcoded dev user via `DevUserMiddleware`). V1 scope is series only (no movies), no Trakt integration yet, no rewatch tracking. Work here follows an unusually rigorous audit-driven style: dry-run before apply, pure-logic modules separated from I/O, real-Postgres integration tests with throwaway fixtures cascade-deleted in `afterEach`, and long narrative markdown docs in `server/docs/` recording *why* decisions were made. Several top-level directories (`library-health/`, `episode-release-refresh/`, `tmdb-enrichment`, etc.) are offline data-migration/audit tools with their own `run-*.ts` CLI entrypoints, not part of the main NestJS app.
- **`mobile/`** (this directory) — Expo/React Native + TypeScript + TanStack Query, React Navigation (bottom-tabs + native-stack). Consumer-facing screens: `HomeScreen` (Watch Next + Recently Watched), `WatchlistScreen` (now two modes — **WATCH LIST** and **UPCOMING**, a chronological personal release timeline, see "Upcoming timeline patterns" below), `LibraryScreen`, `SearchScreen`, `SeriesDetailScreen` (swipe-to-watch and check-button mark-watched, batch watch-all for season/series). There's also an internal migration-workbench admin UI (`MigrationProposalScreen`, `MigrationHistoryScreen`/`MigrationHistoryDetailScreen`, `NeedsAttentionScreen`, `PossibleMatchComparisonScreen`, `ProviderCandidateSearchScreen`) for reviewing/approving the server's stable-version-migration matching decisions.
- **`client/`** — empty placeholder, unused.

For the current cross-cutting server initiative (an objective, batch-capable "Stable Version Migration" policy replacing manual per-title migration decisions — Pipeline A catalog migration in `library-health/`, Pipeline B ongoing refresh in `episode-release-refresh/`), see `server/docs/stable-version-migration-todo.md`, the live authoritative status doc. Don't assume `library-health/`'s uncommitted files or `output/batch*` report directories are experimental/safe to delete without reading it first.

**Upcoming timeline** (the second Shows-tab mode) has its own live worklog with full incident history: `server/docs/upcoming-timeline-todo.md` (Phases 1-10 — design, then three real-device bug rounds post-launch). Read it before touching `UpcomingTimeline.tsx`'s scroll/pagination logic.

**Local dev networking**: `mobile/.env`'s `EXPO_PUBLIC_API_BASE_URL` should point at the Mac's Bonjour/mDNS hostname (`http://<name>.local:3001`, get `<name>` via `scutil --get LocalHostName`), not a raw LAN IP — the IP changes on every DHCP lease renewal and silently breaks "physical device via Expo Go" connectivity (bit this project twice in one session before switching to `.local`). Only fall back to a raw IP (`ipconfig getifaddr en0`) if the network blocks mDNS. Also: `server/`'s default `PORT` fallback (when `.env`'s `PORT` is unset) must stay `3001`, not `3000` — the sibling `nemesh/client` Next.js project defaults to `3000`. **`EXPO_PUBLIC_*` vars are only ever read from `.env`/`.env.*.local` files on disk at build time** (Metro's client-bundle inlining parses them directly, `processEnv: {}` — never from a shell-exported/prefixed `process.env` override, however natural that looks) — see `src/api/config.ts`'s comment before assuming a shell override will work for a one-off test.

MyTV also builds as an installable web PWA (same Expo/react-native-web codebase, exported via `npm run build:web` — see `docs/pwa.md` for the full architecture, manifest rationale, service worker strategy, and known limitations) — deployed as its own Railway service, separate origin from the API. Icons under `web-pwa/icons/` are currently placeholders derived from `assets/icon.png` itself being a placeholder (not real MyTV branding yet); `docs/pwa.md` lists exactly which files to replace once a final logo exists.

# Home / Watch Next patterns

- **Active-tab reselect (scroll to top)**: use React Navigation's own `useScrollToTop(ref)`, not a custom event bus. `Screen.tsx` `forwardRef`s its inner `ScrollView`; each tab-root screen (`HomeScreen`, `WatchlistScreen`, `LibraryScreen`) passes that ref to `useScrollToTop`. `SearchScreen` wires it directly to its `SectionList` ref instead of going through `Screen`. Reuse this pattern for any new tab-root screen — don't invent a parallel mechanism.
- **Watch Next card removal/advance is local-state driven, not cache-driven.** `HomeScreen.tsx` keeps `watchNextOrder`/`watchNextBySeriesId`/`completionState` outside the React Query cache specifically so background refetches (`invalidateQueries`) can't reorder or resurrect a card mid-animation. `markWatchedMutation.onSuccess` sets `completionState` synchronously (brief success state), then `schedulePostWatchReconciliation` (~750ms, `POST_WATCH_RECONCILE_DELAY_MS`) either advances the card in place (real `nextEpisode` in the response) or removes it. Any change to this flow must be verified against a background refetch landing mid-timeout ("stale refetch race") — see `HomeScreen.watchNextReconciliation.test.tsx`.
- **CAUGHT_UP vs COMPLETED**: always branch on `MarkWatchedResponse.userStatus` (server-derived via `deriveUserStatusFromNextEpisode`), never on the older `seriesCompleted` boolean and never render the ambiguous "No more episodes" copy. `CaughtUpCard`'s `outcome` prop expects exactly these two values.
- **Batch watch-all (mark season/series watched) needs no special-casing on mobile.** The server tags batch-created `EpisodeWatch` rows with `watchSource: BATCH` and excludes them from Recently Watched; progress/completion/next-episode calc still see them normally. `runWatchAll` (`SeriesDetailScreen.tsx`) only calls the watch-all endpoints + invalidates `seriesDetail`/`home` — it must NOT insert into `localRecentlyWatched` the way the single-mark path does, or it would defeat the server-side filtering.

# Upcoming timeline patterns

`WatchlistScreen.tsx`'s two modes (WATCH LIST / UPCOMING) **both stay mounted at all times** — switching only toggles RN `display:'none'` on the inactive one, never a conditional unmount. This is what gives free query-cache and scroll-position preservation across mode switches with no extra state-management code — but it also means `UpcomingTimeline` (`SectionList`-based, the first one in this app) can be sitting hidden, with its `useInfiniteQuery` still fetching in the background, while the user looks at Watch List. Three real-device bugs came out of that interaction (full root-cause writeups: `server/docs/upcoming-timeline-todo.md` Phases 8-10) — the lessons, so they don't get rediscovered:

- **A `display:'none'` ancestor never gets a real Yoga layout pass.** Never call `scrollToLocation`/`scrollToIndex` on a list that might be hidden — gate on an explicit `isActive` prop from the parent AND a real `onLayout` event (`hasLaidOut`, set only when the reported height is `>0`), not just "the data is ready."
- **`onStartReached`/`onEndReached` fire immediately on mount** if the list is already "near the start/end" — which it trivially is at scroll offset 0, before any programmatic scroll has run. Combining bidirectional auto-pagination (`SectionList` + `useInfiniteQuery`) with a mount-time `scrollTo`-style anchor requires gating the auto-pagination handlers behind "has the initial scroll already completed" (`canAutoLoadMorePages` in `upcomingGrouping.ts`, keyed off the same `hasAnchoredToToday` ref the anchor effect sets) — otherwise it's a runaway loop: each auto-loaded page still leaves the list at ~offset 0, so it fires again, growing the list far past the intended window before the anchor scroll ever gets a stable target.
- **Never fake `getItemLayout` for variable-height cards/headers** (this app's cards genuinely vary — wrapping titles, optional badges/time/day-count text). Instead: keep the initial window's lookback deliberately small (`UPCOMING_INITIAL_PAST_DAYS` in `upcomingGrouping.ts` — 3, not 14; "a small amount of historical context," not a full lookback) so the target section's index stays low, set `initialNumToRender` above RN's own default (10 → 30 here), and use `onScrollToIndexFailed` with a short, explicitly bounded retry (`canRetryScrollToToday`, capped, never unbounded) as a fallback — not the primary mechanism.
- **Mode-specific tab-reselect uses exactly ONE `useScrollToTop` call**, in `WatchlistScreen` itself, pointed at a dispatcher ref object whose method is reassigned fresh on every render (closes over the current `mode`) and routes to either the Watch List `ScrollView.scrollTo` or `UpcomingTimeline`'s imperative `scrollToToday()` handle (exposed via `forwardRef`/`useImperativeHandle`). Never two independent `useScrollToTop` hooks (one per panel) — that risks firing both simultaneously on a single reselect.
- **Lifecycle/ref-guard decisions are pure, exported functions in `upcomingGrouping.ts`** (`shouldPerformInitialAnchor`, `canRetryScrollToToday`, `canAutoLoadMorePages`, `findTodaySectionIndex`), not inline in the component — same "pure logic separated from I/O" bias the server repo uses throughout, applied here so React-lifecycle timing bugs are unit-testable without mounting a real `SectionList`. Note the real limitation this doesn't cover: none of Phases 8-10's actual bugs were unit-test-catchable even with this pattern, since RNTL's mocked `SectionList` never fires `onLayout`/`onStartReached`/`onScrollToIndexFailed` on its own — these are structurally only reachable on a real device/simulator run.

## RTL/testing gotchas in this codebase

- `render(...)` from `@testing-library/react-native` returns a promise-like — always `await` it.
- Import `act` from `@testing-library/react-native`, not `react-test-renderer` (no TS types installed here, breaks `tsc`).
- Modern RN `ScrollView` exposes `scrollTo` via `useImperativeHandle` on the mounted instance, not on `ScrollView.prototype`. Spying on the prototype silently spies on nothing (0 calls, no error). Capture the real ref (e.g. in a `useEffect`) and spy on that instance.
- `useScrollToTop`'s actual `scrollTo` call is deferred inside a `requestAnimationFrame` callback (lets other `tabPress` listeners call `preventDefault()` first). A plain `fireEvent.press` + `act()` flush won't observe it — await an RAF flush inside the same `act()` block.
- Use `tabBarButtonTestID` on `Tab.Screen` to target tab bar buttons in tests; text-based queries can silently press the wrong element without erroring.
- This RNTL version (14.x) no longer exposes `UNSAFE_getByType`/`UNSAFE_root` for grabbing a real host-component instance to spy on (e.g. a `ScrollView`'s `scrollTo`) — using it throws `TypeError: UNSAFE_getByType is not a function`. Instead, mock the wrapping component (e.g. `jest.mock('../../components/Screen')`) with a minimal `forwardRef` fake that exposes a spy-able method via `useImperativeHandle`; see `WatchlistScreen.tabReselect.test.tsx`.
